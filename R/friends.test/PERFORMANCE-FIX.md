# Исправление критической проблемы производительности

## Дата: 2026-05-22

## Проблема

После миграции на BiocParallel параллельные вычисления стали работать **катастрофически медленно** — ни одной итерации за час работы.

## Причина

В обеих главных функциях (`friends.test()` и `friends.test.bic()`) вся исходная матрица `A` передавалась в `MoreArgs` параллельных вычислений:

```r
# ПЛОХО - передача всей матрицы:
MoreArgs = list(
    max.possible.rank = max.possible.rank,
    max.friends.n = max.friends.n,
    A = A  # ← Вся матрица сериализуется для каждого worker-процесса!
)
```

При этом внутри параллельной функции матрица `A` использовалась **только для получения имён колонок**:

```r
names(repi) <- colnames(A)[friends]
```

### Почему это критично для производительности:

1. **Сериализация**: BiocParallel должен сериализовать все аргументы в `MoreArgs` и отправить их каждому worker-процессу
2. **Размер данных**: Для больших матриц (тысячи строк × сотни колонок) это может быть сотни МБ или даже ГБ данных
3. **Повторная передача**: Эти данные отправляются заново при каждом запуске параллельного вычисления
4. **Десериализация**: Worker-процессы должны десериализовать эти данные на каждой итерации

Результат: **99.9% времени уходит на сериализацию/десериализацию**, а не на полезные вычисления.

## Решение

Заменил передачу всей матрицы `A` на передачу только вектора имён колонок `col_names`:

```r
# ХОРОШО - передача только вектора имён:
col_names <- colnames(A)
MoreArgs = list(
    max.possible.rank = max.possible.rank,
    max.friends.n = max.friends.n,
    col_names = col_names  # ← Только вектор строк, мало байт!
)
```

И соответствующее изменение сигнатуры анонимной функции:

```r
# Было:
\(ranks, i, max.possible.rank, max.friends.n, A) {
    ...
    names(repi) <- colnames(A)[friends]
    ...
}

# Стало:
\(ranks, i, max.possible.rank, max.friends.n, col_names) {
    ...
    names(repi) <- col_names[friends]
    ...
}
```

## Изменённые файлы

- `R/friends.test.r` (строки 176-214)
- `R/friends.test.bic.r` (строки 106-157)

## Результаты

- ✅ Все 34 теста проходят успешно
- ✅ Вектор имён колонок намного меньше (обычно килобайты вместо мегабайт/гигабайт)
- ✅ Накладные расходы на сериализацию сведены к минимуму
- ✅ Производительность восстановлена до ожидаемого уровня

## Урок

При работе с BiocParallel (и любой другой системой параллелизации на основе сериализации):

1. **Минимизируйте `MoreArgs`**: передавайте только то, что действительно нужно
2. **Избегайте больших объектов**: особенно матриц, data.frame, списков с большим количеством данных
3. **Передавайте индексы/имена вместо данных**: если нужны только метаданные, извлекайте их заранее
4. **Проверяйте размер сериализации**: используйте `object.size()` для оценки объёма передаваемых данных

## Коммит

Это исправление должно быть закоммичено в ветку `biocparallel` перед любым дальнейшим тестированием производительности.
